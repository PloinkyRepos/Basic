Overview / bwrap-runner

bwrap-runner

bwrap-runner is the shared Basic sandbox image and local runner for bounded command execution inside an inner bubblewrap sandbox. It is generic local sandbox execution, not a replacement for Ploinky's whole-agent lite-sandbox: true backend, and not an owner of any backend-specific runtime. Research provider agents (e.g. openInterpreterAgent) use the image as their container base and invoke the local sandbox runner inside their own provider containers.

Runtime Shape

The manifest runs through normal Ploinky container execution. The container image is the concrete reference assistos/bwrap-runner:node24-python-bookworm, built from node:24.15.0-bookworm-slim. The image ships bubblewrap, Node.js, a generic Python toolchain (python3, python3-pip, python3-venv, python3-dev, build helpers) so provider agents can prepare Python runtimes compatible with the runner's Python ABI on macOS and Linux hosts. The image deliberately does not install backend-specific Python packages such as Open Interpreter; those live inside provider-owned runtime storage or provider-specific image layers. The default profile exposes BWRAP_RUNNER_IMAGE and runs scripts/build-image.sh as a host preinstall hook so a local Podman or Docker image is built before container creation if missing. Provider manifests that spawn inner bwrap jobs must request containerSecurity.privileged: true because each provider container is itself a sandbox host for inner bubblewrap jobs. The published image installs the reusable local runner at /opt/bwrap-runner/bin/ and /opt/bwrap-runner/lib/, plus an executable shim at /usr/local/bin/bwrap-sandbox-exec. Provider agents invoke that shim or the underlying CLI directly inside their own container; they must not depend on a remote sandbox_exec MCP tool. The published image is produced by manually dispatching .github/workflows/publish-bwrap-runner.yml. It runs on a Linux GitHub runner with Docker Buildx, pushes to Docker Hub for both linux/amd64 and linux/arm64, and must not publish automatically on repository pushes.

Sandbox Policy

The local wrapper accepts only command, stdin, optional staged files, timeoutMs, allowlisted env entries, and optional provider-selected runtime metadata. It rejects extra fields such as raw mounts, binds, capabilities, network selectors, or direct bubblewrap flags. The generated bubblewrap argv is built by bwrap-runner/lib/policy.mjs so policy construction can be tested without invoking bubblewrap.

Each job receives a private state directory under BWRAP_RUNNER_STATE/jobs/<jobId>/. The inner sandbox gets a writable /work, writable /outputs, a tmpfs /tmp, /proc, /dev, and read-only binds for selected system paths that exist in the runner image. When the provider selects a runtime directory, the runner validates it under the provider-owned runtime root, checks manifest identity and symlink containment, and adds a single read-only /runtime bind. The runner never binds /shared or any other portion of the shared directory into inner jobs.

The runner starts the child process with an empty outer environment and lets bubblewrap set only validated environment keys plus manifest-derived defaults such as PYTHONPATH=/runtime/python. Provider credentials and invocation JWTs are never propagated into inner jobs.

Staged files are validated by lib/staging.mjs, decoded from utf8 or base64, and written only below the per-job /work directory before bubblewrap starts. The API deliberately stages data instead of requiring callers to pass generated shell or Node.js driver code in command.

Trust model: /shared is a coordination channel among trusted, explicitly enabled agents in a workspace. It is not the required runtime handoff path for research providers and it does not isolate hostile co-tenant agents. Do not treat the runner as multi-tenant isolation.

Readiness And Limits

Provider startup or readiness should run bin/healthcheck.mjs or the equivalent local probe before reporting research readiness. The probe runs a real nested-bwrap command. If the host kernel or outer OCI runtime blocks nested user namespaces, the provider fails startup or readiness with a structured error and guidance.

The image supplies bwrap, but it cannot change host kernel sysctls, Docker or Podman seccomp policy, AppArmor policy, or rootless runtime namespace rules.

Verification

The focused test suite validates argument policy, environment clearing, timeout limits, staged-file confinement, runtime bundle validation and binding, output tail retention, invalid payload errors visible to MCP callers, manifest startup gates, Basic catalog placement, and the nested-bwrap health probe. The health probe test skips when the local host does not provide /usr/bin/bwrap or nested namespace support.

The normative contract for this image and local runtime is DS002 bwrap Runner Image and Local Sandbox Runtime.
